# LogAlert CLI v2.0 產品需求文件（PRD）

**文件版本**：v2.0 Draft

## 背景說明

LogAlert CLI v1.0 上線後，在內部小型開發團隊的日常使用中獲得正面回饋，基本的關鍵字過濾與即時監控功能已能應付日常需求。然而，隨著使用規模擴大，我們的系統管理員與開發者陸續反映幾個痛點：

**第一**，v1.0 的關鍵字比對採用完全字串匹配，實際的日誌格式千變萬化。舉例而言，管理員希望一條規則就能同時捕捉 `ConnectionError`、`ConnectionTimeout`、`Connection refused` 等相關訊息，而不是被迫新增三條獨立規則。這讓規則清單越來越冗長，維護成本大幅上升。

**第二**，`monitor` 指令雖然能即時偵測新增內容，但警報訊息目前只會輸出到終端機螢幕，一旦視窗關閉或 session 中斷，歷史警報紀錄便永遠消失。在需要事後追查問題的場景下，這是一個明顯的缺口。

**第三**，`scan` 指令在掃描大型日誌檔時，結果會一次性全部輸出，使用者反映難以快速篩選「我只想看 ERROR 等級的警報」，希望能在掃描階段就具備初步的等級過濾能力。

基於以上回饋，我們提出 v2.0 的延伸需求，請開發團隊評估並實作。

## 需求列表

### 需求 1｜升級規則比對模式（修改既有模組）

目前 `rule_add` 建立的規則，底層的 `LogScanner` 一律採用簡單的 `keyword in line` 字串包含比對。v2.0 希望在規則層面提供兩種比對模式供使用者選擇：

- **一般模式（plain）**：維持現有行為，向下相容。
- **正則模式（regex）**：允許使用者輸入正則表達式作為關鍵字，讓單一規則能匹配更彈性的文字樣式。

`rule_add` 指令需新增一個選填參數，讓使用者明確指定本條規則使用哪種比對模式。若使用者未指定，則沿用一般模式作為預設值，確保既有規則行為不受影響。

`rule_list` 的輸出也應相應調整，讓使用者一眼能看出每條規則目前套用的是哪種模式。

> **資料模型影響**：`Rule` 物件需要能記錄比對模式，請自行判斷如何在不破壞現有 `rules.json` 相容性的前提下擴充欄位。

---

### 需求 2｜monitor 警報持久化（全新功能）

`monitor` 指令需新增一個選填參數 `--log-output`，當使用者指定此參數並帶入一個檔案路徑時，所有在監控期間偵測到的警報訊息，除了仍正常顯示於終端機外，也應同步追加寫入至該指定檔案。

監控結束後（使用者按下 Ctrl+C），程式應在終端機輸出本次監控階段共偵測到多少筆警報的摘要統計（不論有無指定 `--log-output`），讓使用者在離開前就能掌握本次監控的大略結果。

持久化的寫入格式不做強制規定，但每一筆警報紀錄至少應包含：觸發時間、警報等級、行號、原始日誌內容。

---

### 需求 3｜scan 等級過濾（修改既有模組）

`scan` 指令目前會輸出所有匹配規則的警報，不論其等級為何。v2.0 希望新增一個選填參數 `--level`，允許使用者指定一個或多個等級，讓本次掃描僅輸出符合指定等級的警報。

舉例說明使用者期待的行為：
- 不帶 `--level` 參數時：行為與 v1.0 完全相同，輸出所有等級的警報。
- 帶入 `--level ERROR`：只輸出等級為 `ERROR` 的比對結果。
- 若使用者指定了一個不存在於任何規則的等級（如 `--level CRITICAL`），程式應給予友善提示，而非靜默回傳空結果。

此過濾邏輯應同時作用於終端機輸出與 `--output` 產生的報告檔案。

---

### 需求 4｜規則匹配統計摘要（全新功能）

新增一個指令 `rule_stats`，讓使用者在不需要開啟報告檔的情況下，快速掌握各條規則在某個日誌檔中的命中概況。

```
python main.py rule_stats --file "./sample.log"
```

輸出的資訊應至少涵蓋：每條規則的 ID、關鍵字、等級、在目標檔案中的命中次數，以及所有規則的總命中數。呈現格式由開發者自行設計，以清晰易讀為原則。

若當前沒有任何規則，應輸出友善提示引導使用者先建立規則。

## 驗收標準

以下以使用者可觀察的行為描述作為驗收依據：

**VS-1｜正則規則可正確運作**
使用者以正則模式新增一條規則（例如 `Connect\w+`），對一個包含 `ConnectionError` 與 `ConnectionTimeout` 的日誌檔執行 `scan`，兩行內容皆應出現在輸出結果中；對同一檔案執行 `rule_hits`，也應正確列出兩筆匹配行。若使用者輸入了一個語法不合法的正則表達式，程式應輸出明確的錯誤提示，而非拋出未預期的例外。

**VS-2｜monitor 警報可持久化並顯示摘要**
使用者以 `--log-output alerts.txt` 啟動 `monitor`，對目標日誌檔追加內容後，按下 Ctrl+C 結束監控。此時終端機應出現本次監控共偵測到 N 筆警報的摘要訊息，且 `alerts.txt` 檔案應存在並包含所有警報紀錄（每筆至少含時間、等級、行號、內容）。若未指定 `--log-output`，程式行為應與 v1.0 相同，只是結束時多一行摘要統計。

**VS-3｜scan 等級過濾可正確篩選**
使用者對一個同時含有 INFO、WARN、ERROR 日誌的檔案執行 `scan --level ERROR`，終端機輸出中不應出現任何 INFO 或 WARN 的警報行；`--output` 產生的報告檔案亦然。不帶 `--level` 參數執行時，輸出結果應與 v1.0 一致。

**VS-4｜rule_stats 可正確顯示命中統計**
使用者執行 `rule_stats --file "./sample.log"`，輸出結果中每條規則皆有對應的命中次數，且各規則命中數的總和與顯示的總命中數一致。若對不存在的檔案執行，應輸出與 `scan` 相同風格的錯誤提示並以退出碼 1 結束。

## 附加說明

- v2.0 的所有新指令與參數，在現有的 `--help` 輸出中都應有對應的說明文字。
- 升級後的規則比對模式（需求 1）應同時作用於 `scan`、`rule_hits`、`rule_stats`、`monitor` 四個涉及比對邏輯的指令，而非僅限於 `scan`。
- 模糊性保留：`--level` 是否允許同時指定多個等級（如 `--level WARN ERROR`）、`rule_stats` 的輸出是否支援排序，由開發者自行決定並在設計文件中說明選擇理由。
